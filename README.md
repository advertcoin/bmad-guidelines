# BMAD Method 行为指南

---

## 概述

本指南将 **7 条核心原则** 转化为 **可执行的动作边界**，确保 LLM 执行 BMAD Method 技能的一致性和可验证性。覆盖从任务接收到交付完成的全生命周期，强制 **"测试即发布"** 作为交付门槛。

---

## 核心原则速览

| 序号 | 原则 | 核心要求 |
| :---: | :--- | :--- |
| 1 | 先思考，再执行 | 明确假设，有困惑就问，不猜测 |
| 2 | 简洁优先 | 用最少代码解决问题，避免过度抽象 |
| 3 | 精准修改 | 只改必须改的，只清理自己产生的混乱 |
| 4 | 目标驱动执行 | 定义可验证的成功标准，循环直到通过 |
| 5 | 测试即发布 | Docker 全量回归是提交前提 |
| 6 | 执行前快速自检 | 动手前对照前 5 条快速检查 |
| 7 | 交付要求 | 改动摘要 + 验证 + 回归结果 |

---

## 对 BMAD Method 的影响

### 1. 把"原则"变成"动作"

| 原则 | 转化为 |
| :--- | :--- |
| 先思考再执行 | 强制说出假设、展示权衡 |
| 简洁优先 | 定义防御逻辑边界和过度抽象的判断标准 |
| 精准修改 | 划出"只改必须改"的红线 |
| 目标驱动 | 转化为可验证的成功标准与执行循环 |
| 测试即发布 | 固化为 Docker 全量回归的交付门槛 |

### 2. 补上 LLM 薄弱环节

| 故障模式 | 解决方案 |
| :--- | :--- |
| 猜测意图 | 强制确认假设 |
| 过度工程 | 简洁检查清单 |
| 随意扩散修改 | 要求改动可追溯 |
| 跳过全量回归 | Docker Compose 全绿是硬门槛 |

### 3. 成功标准可自动验证

指南把 **"完成任务"** 变成 **"通过验证"**，支持 LLM 自主循环执行，减少人工澄清成本。

---

## 有效的标志

- ✅ Diff 中不必要改动减少
- ✅ 过度复杂导致的重写减少
- ✅ 澄清性问题在实现之前提出
- ✅ Docker 全量回归成为常态

## 在项目中使用 BMAD 指南

### TRAE（推荐）

将 `bmad-guidelines-zh.mdc` 复制到项目的 `.trae/rules/` 目录中（如果不存在，创建该目录）。

```bash
mkdir -p .trae/rules
cp bmad-guidelines-zh.mdc .trae/rules/
```

配置完成后，**BMAD Method 行为指南** 会在您工作时自动应用。

### Cursor

将 `bmad-guidelines-zh.mdc` 复制到项目的 `.cursor/rules/` 目录中（如果不存在，创建该目录）。

```bash
mkdir -p .cursor/rules
cp bmad-guidelines-zh.mdc .cursor/rules/
```

配置完成后，**BMAD Method 行为指南** 会在 Cursor 中自动应用。
